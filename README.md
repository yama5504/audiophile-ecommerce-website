# プレミアムオーディオECサイト（マルチページ・カンプ書き起こし実績）

これは [Frontend Mentor の Audiophile e-commerce website](https://www.frontendmentor.io/challenges/audiophile-ecommerce-website-C8cuSd_wx) チャレンジに対する実装コードです。支給された大規模なデザインカンプ（複数画面・データ連携）をもとに、セマンティクスやWAI-ARIA属性を意識したマークアップ、vw やメディアクエリを用いた流体レスポンシブ対応、および Vanilla JS と localStorage を組み合わせた動的なカート機能の実装を行いました。
外部ライブラリやフレームワークに頼らず、ピュアなJavaScript（Vanilla JS）のみを用いることで、データとUIの分離や効率的なイベントハンドリングの丁寧な実装を追求しています。

## 概要

### 要件・機能

4つの異なる画面（トップ、カテゴリ一覧、商品詳細、決済）にわたり、一貫したブランド価値を伝えるデザインをドット単位で忠実にトレース。さらに、以下の高度なフロントエンド機能を実装しました：

- **マルチページ（MPA）ナビゲーション & フルレスポンシブ設計:** スマホ・タブレット・PCの全環境でデザインが滑らかに追従。
- **動的カートシステム（LocalStorage連携）:**
  - 全画面共通のヘッダーからアクセス可能な「カートモーダル」機能。
  - 数量増減、1個未満時の自動アイテム削除、合計金額のリアルタイム同期。
  - 決済画面（`checkout.html`）での動的な注文サマリー（`renderCheckoutSummary`）の自動計算および、注文完了時の「サンクスモーダル（`ThanksModal`）」の制御と購入後のカートクリア。
- **UIコンポーネントの非同期インクルード:**
  ヘッダー、フッター、商品共通バナーなどの使い回すパーツを別HTML化し、JSで動的に挿入するコンポーネント指向の開発スタイル。

## 開発プロセス

### ディレクトリ構造

Viteのマルチページビルド仕様に則り、ルート階層に各画面のエントリーポイントとなるHTMLを配置。ソースコード側は「機能・役割ごとのモジュール（JS/SCSS）」に細かく分離し、最終的に `main.js` と `style.scss` で統合する、クリーンでスケールしやすいディレクトリ構造を設計しています。

```text
.

├── index.html               # メインHTML（トップページ）
├── category.html            # カテゴリ一覧画面
├── product.html             # 商品詳細画面
├── checkout.html            # 決済・チェックアウト画面

├── public/                  # 成果物にそのままコピーされる静的アセット
│   └── assets/              # 商品画像、ロゴ、各種アイコン等のSVG/画像群

    │   ├── cart-ui.js       # カートUI描画・計算・モーダル・サンクス制御（UIロジック）
    │   ├── category.js      # カテゴリ判定・一覧動的生成
```

### 使用技術・環境
- 言語・マークアップ: HTML5 / SCSS (Sass Modules対応) / Vanilla JS (ES6+ オブジェクト指向・モジュール分割)
- ビルドツール: Vite (マルチページ対応構成)
- 自動化環境: GitHub ActionsによるGitHub Pagesへの自動CI/CDパイプライン構築

### こだわった技術的アプローチ
#### 1. JSONデータとURLパラメータによる動的ページ生成

商品の詳細（product.html）やカテゴリ一覧（category.html）は、画面ごとに大量の固定HTMLを作成するのではなく、共通のデータ構造を持つテンプレートHTMLに対して URLSearchParams を用いてパラメータを取得し、data.json から該当データを抽出して動的にDOMを組み立てる設計を採用しました。
データの追加・修正のみでページが生成される仕様を意識し、実務でのCMS連携やJAMstack開発のデータフローを見据えたデータ駆動型のコーディングを行っています。

#### 2. Fetch API と async/await による非同期コンポーネントインクルード

ヘッダーやフッター、パーツなどの「共通部分の二重管理」を防ぐため、Fetch API と async/await を用いた軽量な非同期読み込み関数（component-loader.js）を自作しました。
これにより、HTML側のコード量を削減してメンテナンス性を向上させるとともに、非同期処理の確実なシーケンス（読み込み完了後にUIイベントを初期化する流れ）を担保しています。

#### 3. ローカルストレージと同期するデータの一貫性を意識したカート状態管理

ECサイトの核となるカート機能は、データの変更を行う「純粋なロジック（cart.js）」と、変更を受けてHTMLを操作する「UI表現（cart-ui.js）」に役割を厳密に分離（関心の分離）しました。
データが更新されるたびに localStorage に状態を永続化させ、同時に再描画をかけることで、ヘッダーの個数バッジ、カート内リスト、決済サマリーの3箇所を一律で最新状態へ同期させる確実なデータフローを構築しています。

#### 4. イベント委譲（Event Delegation）によるメモリ最適化とDOMガード

動的に生成されるカート内の商品リストに対して、個別にイベントリスナーを設定するとメモリの浪費や、削除・追加時のリスナー再登録の手間が発生します。
本プロジェクトでは、親要素（.cart-items）にリスナーを1つだけ設定し、e.target.closest('.qty-btn') を用いてクリックされたボタンを動的に判別する「イベント委譲」を採用。さらに、画面ごとに存在しないDOMを操作してJSが停止するのを防ぐ『DOMガード』を随所に配置し、エラーでJSが停止しない丁寧なハンドリングを行っています。

#### 5. <picture> 要素によるブラウザネイティブなマルチデバイス画像最適化

レスポンシブWebデザインにおけるデバイスごとの画像切り替え（モバイル/タブレット/デスクトップ）をJavaScript側でハンドリングするのではなく、HTML5の <picture> および <source media="..."> 要素を用いてブラウザにネイティブ処理させています。
これにより、JavaScriptの計算コストを排除し、デバイスに応じた最適な画像アセットのみをブラウザに最小通信量でダウンロードさせる、Webパフォーマンス（Core Web Vitals）を意識したフロントエンド実装を徹底しました。

#### 6. @use 規則に準拠した Sass モジュール設計（関心の分離）

従来のグローバルな @import によるスタイルの衝突や意図しない上書きリスクを避けるため、モダンな Sass Modules（@use） を全面採用しました。
画面やコンポーネント単位でスタイルファイルを完全に独立させ、メインの style.scss で統合管理。変数やミックスインもカプセル化された名前空間内で呼び出すことで、複数人開発や大規模開発にも耐えうる、安全で予測可能性の高いCSS設計を行っています。

### 今後の拡張計画
入力フォームの総合バリデーション強化:
決済画面（checkout.html）の各入力フィールドに対し、正規表現を用いたリアルタイムなエラーバリデーション（UXを考慮し、フォーカスアウト時にエラーを出す等）の実装を計画しています。

カート内アイテムのローカルストレージ削除ボタン個別実装:
現在は「全削除」と「数量を0にする削除」ですが、カートUI内にピンポイントで「削除」できるボタンを追加し、さらなるUI/UXの向上を図ります。
